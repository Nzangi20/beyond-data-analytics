4.1 Stored Procedures

A stored procedure is a prepared SQL code that you can save, so the code can be reused over and over again. It acts like a function for your database.

Creating Stored Procedures
CREATE PROCEDURE GetAllCustomers
AS
SELECT * FROM Customers
GO;
Executing Procedures
EXEC GetAllCustomers;
Procedures with Parameters
CREATE PROCEDURE GetCustomerByCity @City VARCHAR(50)
AS
SELECT * FROM Customers WHERE City = @City
GO;

-- Execute with parameter
EXEC GetCustomerByCity @City = 'London';
Modifying Procedures

Use ALTER PROCEDURE to modify the logic without dropping and recreating it.

ALTER PROCEDURE GetCustomerByCity @City VARCHAR(50)
AS
SELECT CustomerName, ContactName, City 
FROM Customers 
WHERE City = @City
ORDER BY CustomerName;
GO;

4.2 SQL Parameters

Parameters are used to pass values into stored procedures or dynamic SQL statements. They ensure that user input is treated strictly as data, not as executable code.

Input Parameters

Input parameters allow the caller to pass data into the procedure.

CREATE PROCEDURE UpdateEmployeeSalary 
    @EmpID INT, 
    @NewSalary DECIMAL(10,2)
AS
BEGIN
    UPDATE Employees
    SET Salary = @NewSalary
    WHERE EmployeeID = @EmpID;
END;
GO

-- Execution
EXEC UpdateEmployeeSalary @EmpID = 1, @NewSalary = 85000.00;
Best Practice: Always define the data type and size for parameters to ensure data integrity and prevent overflow errors.

4.3 Prepared Statements

Prepared statements (or parameterized statements) are pre-compiled SQL queries that separate the SQL logic from the data provided.

Benefits
  • Security: The primary defense against SQL Injection. Parameters are treated as literal values, not executable code.
  • Performance: The database compiles the query plan once and reuses it for subsequent executions with different parameters, reducing overhead.
  • Maintainability: Code is cleaner and easier to read when logic and data are separated.
Conceptual Example (Application Side)
-- The application sends the query template and parameters separately
-- Query Template:
SELECT * FROM Employees WHERE Department = ? AND Salary > ?;

-- Parameters passed securely by the application driver:
-- Param 1: 'IT'
-- Param 2: 70000

4.4 SQL Injection

Critical Security Threat SQL Injection (SQLi) is a code injection technique that might destroy your database. It is one of the most common and devastating web hacking techniques.
What is SQL Injection?

SQL injection occurs when malicious SQL statements are inserted into entry fields for execution (e.g., to dump the database contents to the attacker or bypass authentication).

Common Vulnerability Example

If your application constructs queries by concatenating strings directly with user input:

-- VULNERABLE CODE (Conceptual Application Logic)
query = "SELECT * FROM Users WHERE username = '" + userInput + "' AND password = '" + passwordInput + "'";

An attacker could input ' OR '1'='1 as the username, resulting in:

SELECT * FROM Users WHERE username = '' OR '1'='1' AND password = '...'

This condition is always true, granting the attacker unauthorized access!

Prevention Techniques
  1. Parameterized Queries: Always use prepared statements so the database treats user input as data, not executable code.
  2. Stored Procedures: Encapsulate SQL logic and use parameters.
  3. Input Validation: Validate and sanitize all user inputs (e.g., allowlists).
  4. Principle of Least Privilege: Ensure the database account used by the application has only the necessary permissions (e.g., no DROP TABLE rights).

4.5 SQL Comments

Comments are used to explain sections of SQL statements, or to prevent execution of some parts of the code. They are essential for maintainability and collaboration.

Single-line Comments

Start with --. Everything after -- on that line is ignored.

-- Select all employees in the IT department
SELECT * FROM Employees
WHERE Department = 'IT'; -- Filter by IT department
Multi-line Comments

Enclosed between /* and */. Can span multiple lines.

/* 
   Author: Victor Lumumba
   Date: 2026-07-03
   Purpose: Calculate annual bonus for high-performing employees
*/
SELECT FirstName, Salary, (Salary * 0.10) AS Bonus
FROM Employees
WHERE PerformanceRating > 4;

4.6 Data Transfer

SELECT INTO

Copies data from one table into a new table. The new table is created automatically with the same column definitions.

-- Create a backup of all IT employees
SELECT * INTO Employees_Backup
FROM Employees
WHERE Department = 'IT';

-- Copy specific columns to a new table
SELECT FirstName, LastName, Salary INTO HighEarners
FROM Employees
WHERE Salary > 80000;
INSERT INTO SELECT

Copies data from one table and inserts it into an existing table.

-- Insert German customers into an existing CustomerArchive table
INSERT INTO CustomerArchive (CustomerName, City, Country)
SELECT CustomerName, City, Country
FROM Customers
WHERE Country = 'Germany';
Note: SELECT INTO requires appropriate permissions to create new tables in the database.

4.7 SQL Hosting

Database Hosting Concepts

SQL hosting refers to storing your database on a remote server managed by a hosting provider (e.g., AWS RDS, Azure SQL, Google Cloud SQL, or shared hosting like cPanel/MySQL).

Connecting SQL Databases

To connect an application to a hosted database, you typically need a Connection String:

Server=myServerAddress;Database=myDataBase;User Id=myUsername;Password=myPassword;
Deployment Basics
  • Migration Scripts: Use tools like Flyway or Liquibase, or manual SQL scripts to deploy schema changes.
  • Environment Separation: Maintain separate databases for Development, Testing, and Production.
  • Backups: Configure automated daily backups and point-in-time recovery.
  • Firewalls: Restrict database access to specific IP addresses (e.g., only your application server's IP).
